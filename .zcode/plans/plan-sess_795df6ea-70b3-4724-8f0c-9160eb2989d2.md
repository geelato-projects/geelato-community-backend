# 用户委托/助理关系功能 实现方案（修正版）

## 一、需求确认（已与你确认）

| 决策项 | 结论 |
|---|---|
| 关系数量 | **一对多**（一个用户可配置多个代理人） |
| 生效范围 | **业务范围 scope + 启用/禁用状态**（不做时间范围） |
| 消费方式 | **不提供查询接口，其他业务直连表查询** |
| 关系类型 | **不区分类型**，统一为"委托关系"（保留 `relation_type` 预留字段） |

## 二、技术架构对齐（已修正认知）

**后端**（`geelato-community`）：
- 实体放 `geelato-meta/cn.geelato/meta`，参考 `RoleUserMap`（继承 `BaseEntity`、`@ForeignKey`+`@Col(refTables/refColName)`、回填名称用 `isRefColumn=true, refLocalCol, refColName`）。
- Controller/Service 放 `geelato-web-platform/.../srv/security`，沿用 `@ApiRestController`+`BaseController`+`BaseService`、CRUD 命名 `pageQuery/query/get/insert/createOrUpdate/isDelete/validate`、`ApiResult/ApiPagedResult`、`SessionCtx.getCurrentUser()`。
- **不写 init SQL**。表通过实体注解 + 设计时接口 `POST /api/meta/ddl/table/{entity}` 手动触发建/改表（由前端"模型管理"维护）。后端只需交付实体类、Service、Controller 三个 Java 文件。

**前端**（`geelato-front`，pnpm monorepo，Vue3 + Arco）：
- dt（设计时 webapp）= `webapps/gl-admin-arco-dt-std`，rt（运行时 webapp）= `webapps/gl-admin-arco-rt-std`。业务页面都集中在共享包 `packages/gl-admin-core-arco/src`。
- 个人中心路由：`/account/user`、`/account/manage`、`/account/email`，定义在 `packages/gl-admin-core-arco/src/router/routes/modules/account.ts`。每个子页面 = `views/account/<name>/index.vue`（薄壳：Breadcrumb + 业务组件） + `components/account/<name>/index.vue`（业务实现）。
- CRUD 参考：`packages/gl-admin-core-arco/src/views/security/user/{list,form,model}.vue`（用 Arco 原生 `<a-table>` + `<a-modal>`/`<a-drawer>` 表单壳 + 独立 `model.vue` 表单内容）。
- API 封装在 `packages/gl-runtime-core/src/m/datasource/` 下按域分文件（`SecurityApi.ts` 等），通过 `entityApi.getAxios()` 发请求，命名风格 `pageQueryUser/createOrUpdateUser/deleteUser`。
- 用户选择器：现成组件 `GlUserSelect`（`packages/gl-ui-web-arco/src/components/gl-user-select/GlUserSelect.vue`），支持单/多选。
- i18n：每个 component/view 目录下就近 `locale/{zh-CN,en-US}.ts`，通过 `mergeLocaleMessages` 合并。

## 三、详细设计

### 1. 数据库表 `platform_user_r_delegate`（仅设计，不写 SQL）

表名沿用 `platform_<主>_r_<从>` 命名约定（如 `platform_role_r_user`）。字段由实体类注解决定：

| 列名 | 类型 | 说明 |
|---|---|---|
| `id` | varchar(32) PK | 主键（来自 `BaseEntity`） |
| `user_id` | varchar(32) | **委托人**（发起委托的用户） |
| `user_name` | varchar(64) | 委托人姓名（回填冗余字段） |
| `delegate_user_id` | varchar(32) | **代理人**（被配置来代为处理事务的用户） |
| `delegate_user_name` | varchar(64) | 代理人姓名（回填冗余字段） |
| `relation_type` | varchar(32) | 关系类型，默认 `delegate`，预留扩展 |
| `scope` | varchar(500) | **业务范围**，逗号分隔的业务标识（如 `approval,todo`），空表示全部 |
| `enable_status` | int | 启用状态：1启用 / 0禁用 |
| `description` | varchar(500) | 备注 |
| 公共字段 | — | `dept_id/bu_id/tenant_code/del_status/create_at/creator/creator_name/update_at/updater/updater_name/delete_at`（来自 `BaseEntity`） |

**业务唯一性**：Service 层校验（同一委托人 + 同一代理人 + 同一 scope 不重复），不依赖 DB 唯一索引。

**建表方式**：交付实体类后，由你在设计时管理界面登记到 `platform_dev_table`（或直接调用 `POST /api/meta/ddl/table/platform_user_r_delegate`）触发建表。

### 2. 后端新增 3 个 Java 文件

**2.1 实体类** `geelato-meta/src/main/java/cn/geelato/meta/UserDelegateMap.java`
- 仿 `RoleUserMap`：继承 `BaseEntity`，`implements EntityEnableAble`
- `@Entity(name = "platform_user_r_delegate", catalog = "platform")` + `@Title(title = "用户委托关系表")`
- `user_id` / `delegate_user_id` 用 `@ForeignKey(fTable = User.class)` + `@Col(refTables="platform_user", refColName="platform_user.id")`
- 回填 `user_name` / `delegate_user_name` 用 `@Col(isRefColumn=true, refLocalCol=..., refColName="platform_user.name")`
- `enable_status` 用 `@Col(name="enable_status")` + 默认值 `ColumnDefault.ENABLE_STATUS_VALUE`

**2.2 Service** `geelato-web-platform/src/main/java/cn/geelato/web/platform/srv/security/service/UserDelegateMapService.java`
- `@Component @Slf4j`，继承 `BaseService`
- 重写 `createModel` / `updateModel`：
  - 校验不能委托给自己（`user_id != delegate_user_id`）
  - 校验唯一性（同 `user_id` + `delegate_user_id` + `scope` 不重复，排除自身 id 与已逻辑删除记录）
  - 根据 `user_id`/`delegate_user_id` 自动回填 `user_name`/`delegate_user_name`（查询 `platform_user`）
- 提供 `validate(...)` 方法供 Controller 唯一性校验接口使用

**2.3 Controller** `geelato-web-platform/src/main/java/cn/geelato/web/platform/srv/security/UserDelegateMapController.java`
- `@ApiRestController(value = "/security/user/delegate")`，继承 `BaseController`
- 标准接口（完全沿用项目 CRUD 约定）：
  - `POST /pageQuery` — 分页查询（支持按 `user_id`/`delegate_user_id`/`enable_status` 筛选）
  - `GET /query` — 全量查询
  - `GET /get/{id}` — 单条详情
  - `POST /insert` — 新增
  - `POST /createOrUpdate` — 新增或更新
  - `DELETE /isDelete/{id}` — 逻辑删除
  - `POST /validate/{type}` — 唯一性校验（type=`duplicate`）
- 默认过滤 `del_status=0`（由 `BaseService.filterGroup` 自动处理）

## 四、不做的事情（边界）

- **不提供"以他人身份操作/代办登录"能力**。
- **不提供查询接口给其他业务消费**（其他业务直连表查询）。
- **不做时间范围**（无 start_at/end_at）。
- **不写 init SQL / 不写数据库迁移脚本**——表在设计时由实体注解 + `/meta/ddl/table/{entity}` 接口生成。
- **不在系统管理/用户管理里重复暴露**——本次仅在「个人中心」下加「我的委托」入口，语义为"个人管理自己的委托关系"。如你希望管理员也能集中管理所有人的委托关系，告诉我我再加一个 admin 视图。

## 五、实现顺序与工作量

1. 后端 3 个 Java 文件（实体 → Service → Controller）

4. 前端路由与文案登记
